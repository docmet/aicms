## Objective

Add 10 WordPress-specific MCP tools to the MCP server. When a user pastes their WP `mcp_token` into Claude.ai, these tools appear alongside the existing MyStorey tools. The tools route through MyStorey's backend to the user's WordPress site via Application Password auth.

Purpose: This is the core user value — Claude controlling WordPress content via chat.

Output: 10 new WP tools registered in `_build_tools()` and handled in `_dispatch()` / `POST /mcp`.

## Context

@.planning/PROJECT.md

Architecture note: The MCP server's `POST /mcp` endpoint validates `MCPClient.token`. For WP sites the `mcp_token` on `WordPressSite` is the same token — the MCP server calls `GET /api/wordpress/wp-mcp/{token}/dispatch` on the backend to proxy WP REST calls. The backend endpoint looks up the `WordPressSite` by `mcp_token`, instantiates `WordPressClient`, runs the call, and returns the result.

This keeps WP credentials (Application Password) out of the MCP server entirely.

Key file patterns:
- `mcp_server/src/main.py` lines 323–391: `POST /mcp` endpoint with `_dispatch` pattern
- `mcp_server/src/aicms_mcp_server/server.py` lines 870–1473: `_build_tools()` and `_dispatch()` method

1. Add to `backend/src/api/wordpress.py` a new endpoint `POST /wordpress/wp-mcp/{wp_mcp_token}/dispatch`:
   - Path param: `wp_mcp_token: str` (the `WordPressSite.mcp_token`)
   - Body: `{"tool": str, "args": dict}` as a Pydantic model `WPDispatchRequest`
   - NO auth dependency — this endpoint is internal only (called by MCP server, not users). Add an internal secret header check: require `X-Internal-Secret` header matching env var `INTERNAL_SECRET` (or skip check if env var not set, for dev safety).
   - Look up `WordPressSite` by `mcp_token == wp_mcp_token` and `is_active=True`. If not found → 404.
   - Instantiate `WordPressClient(site.site_url, site.app_username, site.app_password_encrypted)`
   - Route `tool` to the appropriate `WordPressClient` method:
     - `"wp_list_posts"` → `list_posts(status=args.get("status", "any"), per_page=args.get("per_page", 20))`
     - `"wp_get_post"` → `get_post(args["post_id"])`
     - `"wp_create_post"` → `create_post(args["title"], args["content"], args.get("status", "draft"), **{k: v for k, v in args.items() if k not in ("title", "content", "status")})`
     - `"wp_update_post"` → `update_post(args["post_id"], **{k: v for k, v in args.items() if k != "post_id"})`
     - `"wp_publish_post"` → `update_post(args["post_id"], status="publish")`
     - `"wp_list_pages"` → `list_pages(status=args.get("status", "any"), per_page=args.get("per_page", 20))`
     - `"wp_get_page"` → `get_page(args["page_id"])`
     - `"wp_create_page"` → `create_page(args["title"], args["content"], args.get("status", "draft"), **{k: v for k, v in args.items() if k not in ("title", "content", "status")})`
     - `"wp_update_page"` → `update_page(args["page_id"], **{k: v for k, v in args.items() if k != "page_id"})`
     - `"wp_publish_page"` → `update_page(args["page_id"], status="publish")`
     - `"wp_list_categories"` → `list_categories()`
     - `"wp_list_tags"` → `list_tags()`
     - `"wp_get_site_info"` → `get_site_info()`
     - `"wp_update_site_settings"` → `update_site_settings(args.get("title"), args.get("description"))`
   - Return `{"result": <WP REST API response>}` on success, `{"error": str}` with 502 on WP error
   - Add Pydantic schema `WPDispatchRequest(tool: str, args: dict = {})` to `backend/src/schemas/wordpress.py`
